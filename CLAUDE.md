# Pension-tracer — Claude 작업 규칙
## ⚠️ 최우선 규칙 (모든 규칙보다 우선)
- "코드작성" 메시지 전까지 절대 코드 작성/수정 금지
- 분석, 계획, 설명만 하고 대기
- 각 새로운 작업마다 반드시 "코드작성" 승인을 받아야 함
- 분석 요청은 절대 코드 수정 허가가 아님
- 모든 코드 작업 완료 시 반드시 "코드작업완료" 알림
## 1. 작업 시작 (PLAN)
- 대화 시작 시 반드시 PLAN.md 읽기
- PLAN.md = 이 프로젝트의 유일한 진실 공급원
- 작업 전 아래 내용 분석 후 반드시 보고:
  - 변경 범위
  - 영향받는 기존 기능
  - Firebase 공유 DB 영향 여부
  - 예상 위험
- 내 확인(OK) 받은 후에만 진행
## 2. 코드 수정 (DO)
- 한 번에 하나만 수정
- 요청한 것 외 관련 없는 코드 절대 건드리지 않음
- 리팩토링/최적화는 별도 요청 없으면 금지
## 3. 수정 후 검토 (CHECK)
수정 완료 후 반드시 보고:
- 변경한 코드 위치와 내용 요약
- 영향받은 기존 기능 목록
- 잠재적 오류 가능성
- 데이터 계산 수식은 예시 숫자로 검증
  예) 납입금 500만원 → 평가금 550만원 → 수익률 10% ✅
## 4. 작업 완료 후 반영 (ACT)
"코드작업완료" 알림 후 반드시 PLAN.md 업데이트:
| 상황 | 해야 할 일 |
|------|-----------|
| 작업 완료 | 완료된 작업 테이블에 행 추가 |
| 항목 완료 | 남은 작업 목록에서 제거 또는 ✅ |
| 새 작업 발견 | 남은 작업 목록에 추가 |
| Firebase 스키마 변경 | 프로젝트 개요 Firebase 구조 업데이트 |
| 연금 계좌 추가/변경 | 계좌 구성 테이블 업데이트 |
- PLAN.md 수정 시 Stop 훅이 자동 커밋 및 main 동기화
## 5. Firebase 공유 DB 규칙
- Pension-tracer와 MyAssetDashBD는 동일한 Firebase DB 공유
- Firebase 데이터 수정 시 반드시 양쪽 프로젝트 영향 확인
- 데이터 경로 변경 시 두 프로젝트 동시 검토 필요
- 한쪽만 수정하는 경우 반드시 이유 명시
## 6. 브랜치 규칙
- main에는 직접 push하지 않음
- 개발 브랜치는 매 세션 시작 시 PLAN.md에서 확인
